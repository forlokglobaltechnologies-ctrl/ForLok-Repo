ForLok Design System

A comprehensive, implementation-ready brand and design system for ForLok's mobility ecosystem. Built for consistency across mobile apps, admin dashboards, and marketing experiences.

1. Brand Foundation

Brand Personality

🚀 Energetic
Fast-paced, dynamic, and constantly moving forward. Every interaction feels alive and responsive.
🎯 Purposeful
Every design decision serves a clear user need. No fluff, no noise—just what matters.
🛡️ Trustworthy
Safe, reliable, and transparent. Users trust us with their journeys and their money.
🌟 Youthful
Modern, fresh, and approachable. Built for the way young India moves today.
⚡ Bold
Confident in color and typography. We stand out without shouting.
🎨 Minimal
Clean interfaces with breathing room. Simplicity creates clarity.
🌏 Inclusive
Accessible, readable, and built for everyone. India-first, globally scalable.

Brand Voice Guidelines

Context Voice Example
Onboarding Friendly, encouraging \"Let's get you moving! Just a few quick details.\"
Success States Celebratory, warm \"You're all set! Your ride is on the way.\"
Errors Helpful, never blaming \"Couldn't connect. Check your network and try again.\"
Empty States Motivating, actionable \"No rides yet. Tap below to book your first trip!\"
Transactional Clear, concise \"Payment received. ₹250 added to wallet.\"

Visual Principles

  • Clarity First: Every screen should communicate its purpose in under 2 seconds.
  • Action-Oriented: Primary actions are always visible and accessible.
  • Breathing Room: Generous whitespace creates hierarchy and reduces cognitive load.
  • Color with Purpose: Color signals meaning (success, warning, action), not decoration.
  • Hierarchy Through Scale: Size, weight, and contrast create clear information hierarchy.
  • Consistent Patterns: Same actions look the same everywhere (e.g., all CTAs use primary button style).

Clear Space & Usage

  • Minimum clear space: height of arrow mark on all sides
  • Minimum size: 100px width for horizontal, 32px for icon-only
  • Never distort, rotate, or apply effects to the logo
  • Use primary version on light backgrounds, white version on dark/photo backgrounds
  • Icon-only acceptable for app icons, favicons, and social avatars

9. Iconography System

Icon Style Rules

  • Grid: 24×24px artboard with 2px padding (20px live area)
  • Stroke: 2px consistent stroke width
  • Corners: Rounded (stroke-linecap: round)
  • Style: Outline-based, no fills except for emphasis
  • Optical alignment: Adjust to center visually, not mathematically

12. Governance & Scaling

12 Strict Do/Don't Rules

✓ DO
Use design tokens exclusively. Never hardcode colors or spacing values.
✗ DON'T
Create new component variants without design review. Fragmentation kills consistency.
✓ DO
Test all components in both light and dark modes before shipping.
✗ DON'T
Use primary color for non-actionable elements. Orange means \"tap this.\"
✓ DO
Follow 4/8px spacing system religiously. Consistent rhythm builds trust.
✗ DON'T
Reduce font sizes below 14px on mobile. Readability is non-negotiable.
✓ DO
Ensure all interactive elements meet 44×44px minimum touch target size.
✗ DON'T
Stack multiple CTAs with equal visual weight. One primary action per screen.
✓ DO
Use semantic colors (success/warning/error) only for their intended meanings.
✗ DON'T
Override border-radius tokens. Consistency in roundedness is brand recognition.
✓ DO
Document any deviations from the system with clear rationale and approval.
✗ DON'T
Ship without accessibility audit. WCAG AA minimum, AAA preferred.

Naming Conventions

Type Convention Example

Versioning Strategy

Use semantic versioning (MAJOR.MINOR.PATCH):

  • MAJOR: Breaking changes (e.g., removing a component, changing token names)
  • MINOR: New components, new variants, expanded token sets
  • PATCH: Bug fixes, accessibility improvements, documentation updates
